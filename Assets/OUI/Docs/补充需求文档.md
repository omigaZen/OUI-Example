# OUI UI管理系统补充需求文档

基于原始需求文档：
<https://github.com/omigaZen/OUI-Example/blob/master/Assets/OUI/Docs/1.%E9%9C%80%E6%B1%82%E6%96%87%E6%A1%A3.md>

本文档用于补充”重复打开同一 Window 时的处理策略”需求，解决当前框架中”Window 已打开或资源加载中时后续打开请求被直接丢弃”的问题。

本期只补充一件事：允许指定 Window 在重复打开时进入排队，等当前窗口关闭后继续按顺序打开。

* * *

## 一、问题背景

原始需求文档中的当前行为是：

- 同类型 Window 为单例
- 若目标 Window 已打开，则重复调用 `ShowUIAsync<T>()` 直接 return
- 若目标 Window 正在加载中，则重复调用 `ShowUIAsync<T>()` 也直接 return
- `Close()` 只关闭显示，不销毁实例

这个行为适合普通功能窗口，但不适合“同一窗口需要连续串行展示多个业务事件”的场景。

典型场景：

- 连续获得多个奖励、宝箱、Buff、结算项
- 业务层连续多次调用 `ShowUIAsync<T>(args)`
- 第一次请求打开成功
- 后续请求因为窗口已打开或正在加载而被直接丢弃
- 结果表现为业务事件缺失，用户只能看到第一条

目标是：

- 保持单例 Window 模型不变
- 默认行为仍保持当前 `return`
- 对少数指定 Window 增加“关闭后继续打开”的排队能力

* * *

## 二、功能补充

### 2.1 重复打开策略

在 `WindowAttribute` 上新增重复打开策略配置。

#### 2.1.1 枚举定义

```csharp
public enum WindowDuplicateShowMode
{
    /// <summary>
    /// 保持当前行为：重复打开时直接return。
    /// </summary>
    Return = 0,

    /// <summary>
    /// 若窗口已打开或正在加载，则缓存参数。
    /// 当前窗口关闭后，按先进先出顺序继续打开。
    /// </summary>
    QueueAfterClose = 1,
}
```

#### 2.1.2 Attribute扩展

```csharp
[AttributeUsage(AttributeTargets.Class)]
public class WindowAttribute : Attribute
{
    public string AssetPath { get; }
    public WindowLayer Layer { get; }
    public bool HideBelow { get; }

    /// <summary>
    /// 重复打开同一Window时的处理方式。
    /// 默认值为Return，保持旧行为不变。
    /// </summary>
    public WindowDuplicateShowMode DuplicateShowMode { get; set; }
        = WindowDuplicateShowMode.Return;
}
```

说明：

- 不新增 `InstanceMode`
- 不新增 `CloseMode`
- 不支持 `Refresh`
- 框架统一假设所有 Window 都是“单例 + Close=Hide”

### 2.2 行为约定

#### 2.2.1 Window 不存在且未在加载

- 按原逻辑执行：加载资源，创建实例，调用 `BindUI()`、`Init()`、`OnOpen(args)`

#### 2.2.2 Window 正在加载中

- 若 `DuplicateShowMode = Return`
  - 直接 return
- 若 `DuplicateShowMode = QueueAfterClose`
  - 不打断当前加载流程
  - 将本次参数加入该 Window 类型的待打开队列
  - 当前加载完成后只展示第一次请求对应的窗口
  - 队列中的后续请求必须等待该 Window 关闭后再依次打开

#### 2.2.3 Window 已存在且已打开

- 若 `DuplicateShowMode = Return`
  - 直接 return
- 若 `DuplicateShowMode = QueueAfterClose`
  - 将本次参数加入该 Window 类型的待打开队列
  - 当前窗口保持不变
  - 当前窗口关闭后，从队列头部取出一组参数重新打开该 Window

#### 2.2.4 Window 已存在但处于关闭状态

- 按原逻辑执行：复用实例并调用 `OnOpen(args)`

### 2.3 队列语义

- 队列按 Window 类型分别维护
- 队列顺序为 FIFO
- 每次只允许同类型 Window 同时显示一个实例
- 自动续开等价于框架内部再次执行一次 `ShowUIAsync<T>(args)`
- 自动续开也必须参与正常的层级排序和 `HideBelow` 重算

### 2.4 参数缓存规则

- 待打开队列中缓存的是调用时刻的 `object[]` 参数快照，而不是外部数组引用本身
- 框架至少需要复制 `object[]` 数组本身，避免外部复用同一数组导致参数被后续修改
- `object[]` 内部元素若为引用类型，是否深拷贝不属于本期范围，保持传引用语义

## 三、边界与清理规则

### 3.1 CloseWindow

- 当 `CloseWindow<T>()` 或 `CloseWindow(Type)` 关闭一个采用 `QueueAfterClose` 的 Window 后：
  - 若该类型待打开队列非空，则自动取出队首参数继续打开
  - 若队列为空，则不做额外处理

### 3.2 DestroyWindow

- `DestroyWindow<T>()` 或 `DestroyWindow(Type)` 被调用时：
  - 销毁窗口实例
  - 释放资源
  - 清空该类型对应的待打开队列
- 销毁优先级高于排队打开，不允许在销毁后再次自动续开

### 3.3 CloseAllWindows

- `CloseAllWindows()` 执行期间，框架需要进入“批量关闭模式”
- 在批量关闭模式下，即使某个 Window 配置了 `QueueAfterClose`，也不得在 `CloseAllWindows()` 过程中自动续开
- `CloseAllWindows()` 结束后，不自动恢复这些待打开队列
- 结果应当是“所有窗口均关闭，且不发生新的自动打开”

### 3.4 DestroyAll

- `DestroyAll()` 执行时：
  - 销毁所有窗口实例
  - 清空所有待打开队列

### 3.5 重复数据与异常

- 默认不对待打开队列做参数去重
- 连续两次完全相同的 `ShowUIAsync<T>(args)` 也应当视为两个独立请求，按顺序展示两次
- 若资源加载失败，该次打开失败；该类型尚未消费的排队请求如何处理，必须在实现中明确
- 本期推荐策略：
  - 当前请求加载失败时清空该类型待打开队列
  - 输出错误日志，避免队列无限积压

## 四、公开接口补充

### 4.1 BaseWindow

`BaseWindow` 公开接口不新增方法，保持原样。

```csharp
public abstract class BaseWindow : MonoBehaviour
{
    public abstract void BindUI();
    public abstract void Init();
    public abstract void Release();
    public virtual void OnOpen(object[] args) { }
    public virtual void OnClose() { }
    public void Close();
    public bool IsClosed { get; }
    public bool HideByAbove { get; }
}
```

说明：

- 重复打开策略属于框架管理行为，不要求业务 Window 自行实现
- 业务层仍只需在 `OnOpen(args)` 中消费参数
- `Close()` 的语义仍然是 `Hide`

### 4.2 OUI

`OUI` 对外接口原则上不新增。

```csharp
public class OUI : MonoBehaviour
{
    public static OUI Instance { get; }
    public IWindowAssetLoader AssetLoader;

    public async UniTask ShowUIAsync<T>(params object[] args) where T : BaseWindow;
    public void CloseWindow<T>() where T : BaseWindow;
    public void CloseWindow(Type type);
    public T GetWindow<T>() where T : BaseWindow;
    public void DestroyWindow<T>() where T : BaseWindow;
    public void CloseAllWindows();
    public void DestroyAll();
}
```

补充语义：

- `ShowUIAsync<T>()` 在 `DuplicateShowMode = QueueAfterClose` 时，不再意味着“窗口已打开则本次调用无效”
- 当目标窗口已打开或正在加载时，该调用可能转化为“进入待打开队列”

如实现方认为有必要暴露更强控制能力，可在后续版本新增以下可选接口，但不属于本期必做范围：

```csharp
public void ClearPendingShow<T>() where T : BaseWindow;
public void ClearAllPendingShows();
public int GetPendingShowCount<T>() where T : BaseWindow;
```

## 五、验收标准

### 5.1 保持兼容

- 未配置 `DuplicateShowMode` 的 Window，行为必须与原始需求保持一致
- 原有直接 `return` 的窗口不需要修改业务代码

### 5.2 队列打开

给某个 Window 配置 `DuplicateShowMode = QueueAfterClose` 后，满足以下行为：

1. 连续调用三次 `ShowUIAsync<T>(A)`、`ShowUIAsync<T>(B)`、`ShowUIAsync<T>(C)`
2. 第一次立即打开，参数为 `A`
3. 第二次和第三次进入待打开队列
4. 第一次关闭后自动打开，参数为 `B`
5. 第二次关闭后自动打开，参数为 `C`
6. 第三次关闭后不再自动打开

### 5.3 加载中排队

当第一次 `ShowUIAsync<T>(A)` 仍在加载资源时，再次调用 `ShowUIAsync<T>(B)`：

- 若 `DuplicateShowMode = Return`，则 `B` 被丢弃
- 若 `DuplicateShowMode = QueueAfterClose`，则 `B` 被排队
- 资源加载完成后先展示 `A`
- 待 `A` 对应窗口关闭后再展示 `B`

### 5.4 全局关闭与销毁

- `CloseAllWindows()` 后，不应有任何排队窗口自动弹出
- `DestroyWindow<T>()` 后，该类型排队数据必须被清空
- `DestroyAll()` 后，所有排队数据必须被清空

### 5.5 HideBelow兼容

- 自动续开的 Window 与普通打开的 Window 一样，必须参与层级排序和 `HideBelow` 重算
- 不允许因为自动续开而跳过遮挡逻辑

* * *

## 六、非目标

本补充需求不包含以下内容：

- 改变“一个 Window 类只有一个实例”的约束
- 支持同类型 Window 多实例并存
- 为不同 Window 配置不同的关闭行为
- 支持 `Refresh`、插队、去重、覆盖旧参数
- 对 `object[]` 参数做深拷贝

本期目标仅为：

- 在保持单例和 `Close=Hide` 模型不变的前提下
- 为指定 Window 提供“重复打开时排队，关闭后继续打开”的能力
